## Core Framework: Optical Media Archive as Queryable Self (OMAQS)

### Definition

**OMAQS** = Treating physical media archives as primary-source identity data requiring database indexing for:
1. Logical metadata (filenames, content, media type)
2. Physical metadata (degradation, labels, storage context)
3. Emotional metadata (tags like "hell", tone, subject matter)
4. Temporal triangulation (cross-referencing dates across media)

### Proposed Schema

```sql
CREATE TABLE optical_media_index (
    id SERIAL PRIMARY KEY,
    label_text TEXT,
    media_type VARCHAR(50), -- CD-R, DVD-R, VHS-C, etc.
    date_label TEXT, -- handwritten date as-is
    date_estimated TIMESTAMPTZ, -- triangulated actual date
    content_type TEXT, -- VHS-C capture, photo dump, backup, etc.
    tags TEXT[], -- 'hell', 'recovery', event names, etc.
    physical_condition TEXT, -- mold, scratches, degradation notes
    storage_context TEXT, -- red fabric, binder, loose, etc.
    photo_path TEXT, -- path to disc photo
    emotional_weight INTEGER, -- 1-10 intensity
    indexable_for_llm BOOLEAN DEFAULT true
);
```

**Schema reveals assumptions:**
- **Physical degradation is metadata, not failure**
- **Handwritten labels preserved as-is** (not corrected)
- **Emotional weight** is quantifiable (1-10 scale)
- **LLM indexability** as explicit field (some memories may be too raw)

### GPT-4o's Three-Layer Approach

**1. Physical Metadata Layer**
- Photo of each disc (front, back, degradation details)
- Label transcription (exact text, including misspellings)
- Storage context (binder, fabric, location found)

**2. Logical Metadata Layer**
- File listings (if disc is readable)
- Content structure (folder hierarchy)
- Media types (AVI, MPG, JPG, etc.)

**3. Emotional Metadata Layer**
- Tags ("hell", "recovery", "event x", person names)
- Intensity rating (1-10)
- Subject matter keywords
- LLM indexability flag (some too raw to expose to AI)
